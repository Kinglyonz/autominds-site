01 — Infrastructure

What Was Built From Scratch

01

Dedicated Scraper Infrastructure (AWS EC2)

  • Stood up a dedicated EC2 instance (t3.micro, us-east-1) purpose-built for automated procurement scraping
  • Configured headless Chrome (Puppeteer) with optimized memory flags for reliable operation on low-cost infrastructure
  • Built a shared browser launcher module used by all scrapers — standardized user agents, viewport settings, session profiles
  • All scrapers follow a consistent architecture: scrape → normalize → upsert to PostgreSQL (no duplicates)

03

eMMA CAPTCHA Bypass (Maryland State Procurement)

Maryland's eMMA platform is protected by Google reCAPTCHA Enterprise — this blocks every standard scraping approach. Engineered a pure HTTP + 2Captcha integration that solves the CAPTCHA programmatically at ~$0.003 per solve.

  • No browser needed — clean HTTP sessions with cookie management
  • Includes 3-attempt retry logic with fresh sessions on failure
  • Non-trivial engineering challenge that most scrapers can't handle
2Captcha API reCAPTCHA Enterprise HTTP Sessions Cookie Management
04

PostgreSQL Database (AWS RDS)

  • Provisioned and configured a dedicated RDS PostgreSQL instance for SLED data
  • Designed a normalized schema with 20+ fields per opportunity (title, description, agency, value, dates, contacts, documents, state, source URL, etc.)
  • Built upsert logic (INSERT ON CONFLICT) so daily scrapes never create duplicates — they update existing records
  • 530+ records and growing daily
AWS RDS PostgreSQL Upsert Logic
05

Full Frontend Integration (SLED Bridge)

  • Built server/sled-db.ts — PostgreSQL connection pool with parameterized query builder supporting search, source, state, agency, status, pagination
  • Added REST API endpoints: GET /api/sled/opportunities and GET /api/sled/sources
  • Implemented dual-query architecture — federal (GraphQL) and SLED (REST) queries fire in parallel for zero wait-time penalty
  • Added "Data Source" filter: All Sources / Federal (SAM.gov) / State & Local
  • Added "State/Region" filter: All / DC / MD / VA / Federal — wired end-to-end
TypeScript REST API GraphQL Parallel Queries
06

Source Badges on Opportunity Cards

Every SLED opportunity now shows a color-coded badge indicating its source. Users can immediately identify provenance and click through to the original listing.

07

Cross-Source Deduplication

When "All Sources" is selected, the same opportunity can appear from multiple sources (e.g., a Fairfax bid on both BidNet and the county scraper). Built client-side dedup by solicitation number — SLED direct source is preferred over aggregator data. Prevents duplicate cards from cluttering results.

08

Opportunity Detail View — Source Links

  • SLED opportunities in the detail modal show a direct "View Original Listing" link back to the source portal (eMMA, BidNet, eVA, etc.)
  • Source badge displayed in the header for data provenance
  • Contact info, documents, and location data all carried through from the scraper
09

Automated Daily Scraping (Cron)

  • All 7 scrapers run automatically at 6:00 AM ET every day via cron
  • Results logged to a structured JSON file (cron-log.json) with per-source metrics: scraped count, inserted/updated counts, error details, duration
Cron Structured Logging JSON Metrics
10

Scraper Health Monitoring

  • Built and deployed monitor.js — runs daily at 7:30 AM ET (90 minutes after the scrape)
  • Checks every source for: zero-result runs, stale data (25h+), high error rates (>30%)
  • Reports status as CRITICAL / WARNING / OK with detailed per-source breakdown
  • SNS alerting integration ready — just needs a topic ARN for email/SMS alerts on failures
